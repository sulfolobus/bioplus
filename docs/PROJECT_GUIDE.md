## 一、规范目标

本规范用于统一以下工作：

1. 内容新增与改版流程
2. 中英文镜像维护流程
3. 导航与配置维护流程
4. 评论系统与发布流程
5. 交接与质量检查流程

---

## 二、项目结构规范

### 2.1 顶层目录规范

```text
bioplus/
├── docs/                      # 中文内容根目录
├── docs/en/                   # 英文内容根目录
├── overrides/                 # MkDocs 主题覆写目录（含 giscus 注入模板）
├── turtle-knowledge-base/     # 原始提取文本库
├── .github/workflows/         # 自动部署流程
├── mkdocs.yml                 # 中文配置
├── mkdocs-en.yml              # 英文配置
└── *.py                       # 内容处理脚本
```

### 2.2 内容域规范

当前站点分为三个内容域：

1. 研究领域（Research Areas）
2. 学习资源（Learning Resources）
3. 中医经典/Traditional Chinese Medicine

### 2.3 学习资源三级入口规范

学习资源下必须维持以下 3 个一级专题：

1. 高中生物学知识
2. 生物学竞赛知识
3. 高等教育中的生命科学

新增学习专题时，需先评估是否并入上述三级体系，避免无序扩张。

## 三、命名与路径规范

### 3.1 中文内容命名

- 中文目录: 使用自然中文语义
- 中文文件: 使用中文标题（含必要章节序号）
- 示例: `高中生物学知识/必修一/第1章 走近细胞.md`

### 3.2 英文内容命名

- 英文目录: kebab-case
- 英文文件: kebab-case
- 示例: `high-school-biology/compulsory-1/chapter-1-introduction-to-cells.md`

### 3.3 路径映射要求

中文每新增一个结构化页面，必须明确其英文映射路径并更新导航。

---

## 四、页面内容规范

### 4.1 页面 Front Matter 规范

评论区默认全站自动开启；仅在特殊页面关闭：

```yaml
---
comments: false
---
```

### 4.2 标题层级规范

- 每页只允许一个 `# 一级标题`
- 主体章节使用 `##`
- 子级说明使用 `###`

### 4.3 引用规范

- 优先权威来源（高校教材、公开课程、官方机构）
- 禁止使用不明确来源的转载文本作为核心证据
- 建议在页尾设置“参考资料/References”

### 4.4 学习资源模板规范

学习资源类页面建议包含以下块：

1. 学科定位
2. 章节框架/知识点清单
3. 公开学习资源（教材、MOOC、公开课）
4. 学习建议或训练建议

---

## 五、导航维护规范

### 5.1 导航修改原则

任何新增/重命名/删除页面，必须同步更新：

1. `mkdocs.yml`
2. `mkdocs-en.yml`

### 5.2 导航层级原则

- 一级栏目用语义分组（研究领域、学习资源等）
- 二级栏目为专题总览
- 三级及以下为具体主题页

### 5.3 YAML 书写规范

- 使用 2 空格缩进（保持一致）
- 避免混合 Tab
- 修改后必须执行配置检查与本地预览

## 六、中英文同步规范

### 6.1 同步触发条件

以下任一变更会触发中英文同步：

1. 新增栏目
2. 调整目录层级
3. 首页入口改版
4. 导航结构变更

### 6.2 同步清单

每次变更需至少检查：

1. 中文页面已创建
2. 英文镜像页面已创建
3. 中英导航均已更新
4. 首页入口中英均可访问

### 6.3 不同步例外

若页面明确为“中文独有资料页”，需在提交说明中写清原因与范围。

## 七、评论系统维护规范（Giscus）

### 7.1 统一配置位置

评论系统参数统一维护在：

- `mkdocs.yml` -> `extra.giscus`
- `mkdocs-en.yml` -> `extra.giscus`

### 7.2 模板注入位置

全站注入模板在：

- `overrides/main.html`

### 7.3 参数变更规范

如需调整仓库、分类或映射规则，必须同时更新中英文配置并验证。

### 7.4 页面级开关

默认开启评论；单页可通过 `comments: false` 关闭。

## 八、脚本与资料库规范

### 8.1 脚本职责

- `extract_turtle_docs.py`: 提取原始资料文本
- `fix_references.py`: 批量修正引用格式
- `create_en_pages.py`: 批量生成英文占位页（特定场景）

### 8.2 执行要求

- 执行脚本前先确认输入目录
- 执行后复核输出内容，禁止直接无审查发布
- 批处理修改后必须抽样检查页面质量

### 8.3 原始资料库

`turtle-knowledge-base/` 仅作资料和中间产物，不直接作为导航发布内容。

## 九、开发与预览规范

### 9.1 本地预览

```bash
cd /Users/wy/Documents/bioplus
source .venv/bin/activate
python3 -m mkdocs serve
```

### 9.2 构建检查

```bash
python -m mkdocs build -f mkdocs.yml --strict
python -m mkdocs build -f mkdocs-en.yml --strict
```

### 9.3 验收标准

1. 构建无报错
2. 导航可正常点击
3. 新增页面可访问
4. 评论区在非首页正常显示

### 9.4 提交前自动检查命令清单

每次执行新的页面生成或内容修改后，提交前按顺序执行以下命令：

```bash
source .venv/bin/activate
python -m mkdocs build -f mkdocs.yml --strict
python -m mkdocs build -f mkdocs-en.yml --strict
python -m mkdocs serve -f mkdocs.yml -a 127.0.0.1:8000
python -m mkdocs serve -f mkdocs-en.yml -a 127.0.0.1:8001
git status --short
```

说明：

1. `serve` 用于人工抽样检查页面可访问性和评论区加载，检查完成后结束进程。
2. 若只做非交互检查，可跳过 `serve`，但必须保留双语 `build --strict`。

## 十、发布规范

### 10.1 标准发布流程

```bash
git add -A
git commit -m "docs: update generated pages and synced navigation"
git push origin main
```

### 10.2 自动发布说明

push 到 `main` 后，GitHub Actions 自动执行构建并发布。

### 10.3 发布后检查

- Actions 流程状态
- 中文站首页与核心栏目
- 英文站首页与核心栏目
- 新增页面与评论区是否生效

### 10.4 自动提交与推送规则

1. 每次执行新的页面生成或内容修改后，必须在检查通过后自动执行提交并推送。
2. 提交应仅包含当前任务相关改动，避免混入无关文件。
3. 提交信息需明确说明变更类型与范围（如 docs、nav、config、rules）。
4. 默认推送目标为 `main`；若有用户指定分支，以用户要求为准。
5. 仅在“用户明确要求不推送”或“远程鉴权/网络失败”时允许暂不推送，并需在结果中说明原因与后续命令。

## 十二、交接规范

### 12.1 新成员最低准备

- 了解目录结构
- 了解双配置构建模型
- 能独立完成“新增页面 -> 更新导航 -> 本地预览 -> 提交发布”

1. `PROJECT_DOC.md`
2. `PROJECT_GUIDE.md`
3. `mkdocs.yml`
4. `mkdocs-en.yml`
5. `.github/workflows/deploy.yml`

- 严禁只改中文不改英文（除非明确声明）
